## [Unreleased]

### Added

- `domino_cli` crate.
- Cloud-init installs `cloudflared` from Cloudflare's apt repo and the `cloudflared.service` systemd unit, but does **not** start the service or bake any Cloudflare-side secret into the droplet image. The unit's `ConditionPathExists=` gates on `/etc/cloudflared/cert.pem` and `/etc/cloudflared/config.yml`, both written post-boot by `automation/scripts/setup-tunnel.sh` from the Pulumi-canonical secrets (see the `Pulumi.heart.yaml` entry below). The service runs as `root` (not `DynamicUser`) because the per-tunnel credentials file has to live at a stable persistent path that survives restarts. HA fleet behaviour comes from running the same credentials file on every always-online droplet — Cloudflare load-balances across connectors registered against the same tunnel id.
- Cloud-init installs the `hc-http-gw` systemd unit, the `/etc/hc-http-gw/env` template, the `hc-http-gw-launcher` wrapper, and the idempotent `/usr/local/bin/hc-http-gw-configure` helper. The binary itself is installed post-boot by the operator via [`unytco/automation`](https://github.com/unytco/automation)'s `setup-gateway.sh` (Makefile target `heart-always-online-N-gateway`), which clones `holochain/hc-http-gw` at `.gateway.version` and runs `cargo build --release` on the droplet (the heart-side mirror release plan was abandoned, and there's no `workshop/hc-http-gw` submodule today — see `doc/upstream-hc-http-gw-release-todo.md`). The systemd unit's `ConditionPathExists=` gates on all three required files (binary + launcher + env), so cloud-init succeeds while the binary is absent; the gateway comes online once the operator runs `setup-gateway` + `hc-http-gw-configure`.
- Optional Pulumi `digitalocean.Firewall` (`heart-fleet-firewall`) attached by tag to every droplet flavour. Inbound: SSH from a `heart:operator-cidrs` allowlist + ICMP. No inbound 8090/8800/80/443 — `cloudflared` is outbound-only. Firewall is skipped entirely unless `heart:operator-cidrs` is set, so operators can't lock themselves out by accident.
- `/etc/heart-fleet/metadata` exposes the public gateway hostname (`HEART_GATEWAY_HOSTNAME`, sourced from the new `heart:gw-hostname` Pulumi config) on every droplet so operator one-liners in `README.md` and `doc/setup-always-on-node.md` can `source /etc/heart-fleet/metadata && curl -i "https://${HEART_GATEWAY_HOSTNAME}/health"` without hard-coding the value.
- `scripts/smoke-test.sh <hostname> [...]` exercises the full chain (CF worker → tunnel → gateway → conductor) from anywhere; suitable for manual ops or CI after `pulumi up`.
- `scripts/sync-cloud-configs.sh` enforces byte-identity between `cloudinit/default/` and `cloudinit/alt/`. Run it after any cloud-config edit; `--check` mode is CI-friendly.
- `Pulumi.heart.yaml`: new encrypted entries `heart:cf-cert-pem` and `heart:unyt-tunnel-credentials-json` (base64-encoded contents of the CF account origin cert + the `unyt-tunnel` tunnel's credentials JSON). Pulumi is now the canonical, team-accessible source of truth for the secrets the locally-managed cloudflared flow consumes; operators materialize them onto their laptops via `automation`'s `make pull-secrets`. Bitwarden remains as a break-glass DR backup. See [`automation/docs/hash-explorer-backend.md`](https://github.com/unytco/automation/blob/main/docs/hash-explorer-backend.md) § Secrets for the rotation flow and § "0b. One-time tunnel bootstrap" for how the secrets first got seeded into the stack.

### Changed

- bump cloud-init HOLOCHAIN_VERSION and holo-keyutil deps to Holochain 0.6.1.
- conductor admin `allowed_origins` tightened from `*` to `"hc-http-gw,hc-sandbox"`. Admin port stays loopback-only; this is defence-in-depth for the in-process clients we actually use.
- Retired `cloudflare.go` (Pulumi-managed tunnel resources) and dropped the `pulumi-cloudflare` SDK dependency. The locally-managed tunnel model (cert.pem + per-tunnel credentials.json on each droplet, distributed by `automation/scripts/setup-tunnel.sh` from Pulumi-canonical secrets) was the actual production cutover path; the Pulumi-managed `ConfigSrc=cloudflare` flow was never adopted in production. `renderCloudInit` is now a plain `string` return (no `pulumi.StringOutput` / `ApplyT` plumbing) since the remaining template substitutions (`InfluxToken`, `GatewayHostname`) are both regular Pulumi config strings. The hostname previously derived from `provisionCloudflareTunnel()` now comes from a regular `heart:gw-hostname` config key (set to `unyt-tunnel.unyt.co` for the production stack). If we ever want to revisit the Pulumi-managed flow, `git log --follow -- cloudflare.go` recovers the working code.
- Abandoned a heart-side mirror of upstream `hc-http-gw` releases (previously drafted as `.github/workflows/release-hc-http-gw.yml`). Mirroring a binary we don't author into a private heart release was the wrong supply-chain shape: it would have signed bytes with the wrong key, required GitHub auth on every droplet, and pulled binary-distribution responsibility into a repo that doesn't own the source. The binary is now installed by `automation/scripts/setup-gateway.sh` which (until upstream binaries exist) clones `holochain/hc-http-gw` directly at `.gateway.version` and runs `cargo build --release` on the droplet. Other unytco-owned binaries (`lair-sign`, `pricing-oracle`) flow through `workshop/` submodule pins + `scp`; migrating `hc-http-gw` to the same shape is tracked in `unytco/TODOS/upstream-hc-http-gw-binaries.md` alongside the upstream-PR plan. See [`doc/upstream-hc-http-gw-release-todo.md`](doc/upstream-hc-http-gw-release-todo.md) for the closer-look at the upstream gap.
- Cloud-init now installs `/usr/local/bin/hc-http-gw-launcher` and the `hc-http-gw.service` unit's `ExecStart=` points at the launcher instead of the binary directly; `EnvironmentFile=` is dropped. Required because `hc-http-gw` env var names can include hyphens / dots / version suffixes (e.g. `HC_GW_ALLOWED_FNS_unyt-sandbox-holo-hosting-0.90`) which systemd's `EnvironmentFile=` parser silently drops as "invalid assignment", causing `Missing HC_GW_ALLOWED_FNS_<id> env var` at startup. The launcher reads `/etc/hc-http-gw/env` permissively and execs the binary via `/usr/bin/env`, which uses `execve()` directly and accepts arbitrary key strings. The unit now `ConditionPathExists=`-gates on all three required files (binary, launcher, env). Confirmed live during the `heart-always-online-3` cutover; same fix mirrored into `automation/scripts/setup-gateway.sh`. The launcher body is kept byte-identical with automation's runtime install path; both source files carry a ⚠️ CROSS-REPO COUPLING note pointing at the other, and `automation/scripts/check-launcher-drift.sh` verifies the invariant on demand (see `automation/docs/hash-explorer-backend.md` § "Cross-repo drift checks").
- `/etc/hc-http-gw/env` is now written with mode `0644` (was `0640`) in both the initial cloud-init `write_files` entry and the `hc-http-gw-configure` helper. The `DynamicUser` systemd creates for `hc-http-gw.service` isn't in the root group, so `0640 root:root` yields EACCES at startup (confirmed during the cutover — the launcher's `not readable` stderr was the smoking gun). The env file holds no secrets — just a loopback URL, app ids, and function allowlists, which are intentionally HTTPS-exposed posture anyway — so `0644` is correct.
